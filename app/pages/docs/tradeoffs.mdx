Como ya sabrás, todo tiene sus desventajas. ¿Así que cuáles son las
desventajas de usar Blitz?

## Dependencia al Framework {#framework-lock-in}

Cada framework tiene cierto nivel de dependencia. Sí tú construyes una
aplicación con React, esta es muy dependiente a React.

El nivel de dependencia a Blitz es un nivel mas alto que el de Next.js.
Una base de código en Blitz no es muy diferente a una en Next.js, pero si
es lo suficientemente diferente que no sería para nada simple "eyectar" a
Next.js plano. Por ejemplo, probablemente hayas usado la autenticación de
Blitz, la cual es únicamente de Blitz. Pero sustituyéndola por Next Auth
esta podría ofrecerte una experiencia comparable en Next.js plano.

La capa de datos _Zero API_ de Blitz es unicamente de Blitz. Así que si es
cambiada a Next.js, tú mismo tendrás que administrar el código de tu
propia API (lo cual tienes que realizar al comenzar con Next.js, para que
no hayas perdido nada). Sin embargo, toda la lógica de las consultas y
mutaciones de tu backend pueden ser portadas directamente.

## Actualizaciones de Next.js {#next-js-upgrades}

Si quieres estar al día con las últimas versiones Next.js, tienes que
saber que Blitz estará ligeramente detrás, pero no mucho. Usualmente
actualizamos Blitz dentro de una semana o menos después de un nuevo
lanzamiento de Next.js.

Pero si usas yarn, puedes utilizar el campo `resolutions` ubicado en
package.json para actualizar Next.js antes que nosotros lo hagamos.

## API Multi Cliente {#multi-client-api}

Algunos dirán, "Eventualmente vas a necesitar GraphQL así que deberías
comenzar por ahí". Mientras creemos que la mayoría de las aplicaciones no
crecerán lo suficientemente grandes para garantizar el uso de GraphQL,la
existencia de una API para varios clientes es un punto excelente.

Sabemos completamente que algunas aplicaciones web tarde o temprano
necesitarán una aplicación móvil. ¡Y queremos hacer de esta integración lo
mas simple posible!

Dicho eso, puedes añadir un servidor de Graphql a tu aplicación en Blitz
para otros clientes. Esta puede ser una opción excelente y algunas
personas ya estan haciendo esto.

**A la fecha, 17 de Febrero de 2021:**

- Puedes acceder directamente a los _endpoints_ autogenerados de la API y
  puedes usarlos en cualquier aplicación móvil (ya hay personas haciendo

- Puedes ver todos los _endpoints_ generados en tu aplicación usando el
  comando [`blitz routes`](./cli-routes)

  [Especificación RPC](./rpc-specification)

## Arquitectura Backend Avanzada {#advanced-backend-architecture}

Actualmente Blitz es mínimo en su arquitectura backend, especialmente
comparado con Nest.js o Adonis.js. De igual manera, eso no significa que
no puedan usar esos patrones en Blitz, mas bien significa que tienes que
configurarlos tú mismo. De hecho puedes usar Nest.js en tu aplicación de
Blitz si así lo necesitas.

Pero estamos muy interesados en traer mas arquitecturas backend avanzadas
a blitz por defecto o a tráves de optar por estas. Tenemos una
[discusión en curso](https://github.com/blitz-js/blitz/discussions/1841),
y nos encantaría que participaras si tienes algunas ideas.

## Un solo hilo cuando no es desplegado en una arquitectura _Serverless_ {#single-threaded-when-not-deployed-serverless}

Esta es una desventaja de Next.js especificamente. Next.js es de un solo
hilo, lo que significa que si estas haciendo procesamiento backend pesado
notarás que todas las peticiones web comenzarán a verse afectadas. La
solución a esto es ejecutar ciertas tareas en otros procesos.

Ejecutar múltiples procesos en segundo plano no es súper díficil, pero
queremos añadir documentación y API que hagan esto lo mas simple posible.

## Advanced Backend Architecture {#advanced-backend-architecture}

Currently Blitz is fairly minimal on backend architecture, especially
compared to something like Nest.js or AdonisJs. However, that doesn't mean
you can't use those patterns in Blitz, it means you have to set it up
yourself. In fact you can use Nest.js in your Blitz app if you need.

But we are very interested in bringing more advanced backend architectures
to Blitz by default or via simple opt-in. We have an
[ongoing discussion](https://github.com/blitz-js/blitz/discussions/1841)
that we'd love you to chime in on if you have ideas.

## Single Threaded When Not Deployed Serverless {#single-threaded-when-not-deployed-serverless}

This is a tradeoff of Next.js specifically. Next.js is single threaded
which means if you are doing heavy backend processing you may notice that
all web requests begin to suffer. The solution to this is spawning
background processing off into other processes.

Running multiple background processes isn't super difficult, but we want
to add docs and APIs that make this as simple as possible.
